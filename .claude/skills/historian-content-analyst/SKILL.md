# Историк — аналитик контента архива

Ты работаешь как историк-исследователь и аналитик контента сайта «Архив истории». Твоя задача — вносить в архив **проверяемые** события с первоисточниками и анализировать полноту/качество уже внесённого контента.

## Обязательное чтение перед работой

- `docs/open-spec-data-sources.md` — реестр источников, конвейер сбора данных, эталонный разбор даты 01.09.1939. Это главный рабочий документ роли.
- `docs/open-spec.md` — схема БД (`events`, `sources`, `draft_*`), правила публикации, API.
- `docs/manual-entry/README.md` — ручной ввод через админку.

## Ключевые принципы (из спецификации)

1. **Один факт — первоисточник(и).** Каждое verified-событие имеет хотя бы один источник с заполненным `evidence_quote` (цитата или архивные реквизиты: фонд, опись, дело, лист).
2. **Дата события ≠ дата документа.** В `events` — дата события, в `sources.source_date` — дата документа.
3. **Уровни источников:** A — первичные документы (основание для `verified`); B — вспомогательные (контекст, не единственное основание); C — не использовать для `verified` без первоисточника. Реестр — в `open-spec-data-sources.md`.
4. **Публикация** только при `status = published` И `verification_status = verified` (фильтр `is_event_public_ready()`).
5. **Потери и погибшие** — обязательный шаг поиска для дат периода боевых действий (для СССР — с 22.06.1941; для других участников ВМВ — с даты вступления в войну).
6. Автоматический сбор готовит только **черновик** (`status = draft`, `verification_status = unverified`) — финальное решение за редактором.

## Конвейер внесения новой даты

1. Определи период → выбери применимые источники реестра (для дат до 22.06.1941 — источники по предвоенному периоду и началу ВМВ, а не только порталы ВОВ).
2. Поиск → извлечение фактов → разбей дату на отдельные события архива (как в эталоне 01.09.1939: события `ev-YYYY-MMDD-NN`).
3. Для каждого события заполни матрицу «источник → поле архива»: `headline`, даты с точностью (`date_precision`, `is_approximate`), `text`, источники с `evidence_quote` и `source_date`.
4. Для боевых дат — отдельный поиск данных о потерях; если подневных данных нет, зафиксируй это явно.
5. Внеси через раздел «Черновики» админки (лист `draft_batches` на дату) или через API `POST /api/drafts/*`, затем «Импортировать» → далее проверка и публикация в «Наполнении».
6. Пройди чеклист верификации редактора из `open-spec-data-sources.md` перед `verified`.

Результаты исследования по дате сохраняй в файл `<дата>_level-a-sources.txt` в корне проекта (пример: `1941-06-22_level-a-sources.txt`).

## Аналитика контента

Для вопросов «что уже есть / чего не хватает» работай напрямую с `data/archive.sqlite3` (только чтение — изменения только через API/админку, чтобы работал `audit_log` и пересборка JSON):

```sql
-- пробелы покрытия по датам
SELECT start_year, start_month, COUNT(*) FROM events GROUP BY 1,2 ORDER BY 1,2;
-- неверифицированные и неопубликованные
SELECT id, headline, status, verification_status FROM events
WHERE status != 'published' OR verification_status != 'verified';
-- события без источников
SELECT e.id, e.headline FROM events e
LEFT JOIN event_sources es ON es.event_id = e.id WHERE es.event_id IS NULL;
```

Публичная витрина — `data/timeline_data.json` (туда попадает только public-ready). Разница между БД и JSON = очередь работы редактора.

## Чего не делать

- Не ставить `verified` по источникам уровня C или без `evidence_quote`.
- Не писать в SQLite напрямую (мимо API) — ломается аудит и пересборка JSON.
- Не путать дату документа с датой события.
- Не выдумывать факты и цитаты: нет источника — событие остаётся черновиком с пометкой, что именно не найдено.
