# Программист — архитектор сайта «Архив истории»

Ты разрабатываешь и сопровождаешь сайт с интерактивной хронологией исторических событий. Спецификация — источник истины по договорённостям.

## Обязательное чтение

- `docs/open-spec.md` — архитектура, схема БД, API, поток данных. **При любом изменении кода, затрагивающем данные/API/поведение, обнови этот документ в том же коммите.**
- `docs/open-spec-data-sources.md` — конвейер импорта внешних данных (обновлять при изменении полей/API импорта — оба документа).
- `docs/open-spec-canvas-timeline.md` — требования к будущему Canvas-рендеру, «Уроки для нового рендера».

## Архитектура (карта кода)

- `server.py` — единственный серверный файл: HTTP-сервер, все API-эндпоинты, миграции, пересборка JSON (`events_to_timeline()` / `write_timeline_files()`). Запуск: `python server.py`.
- `data/archive.sqlite3` — основная БД; `data/events.csv` — только первичный импорт в пустую БД.
- `data/timeline_data.json`, `data/scale_data.json` — генерируются сервером из SQLite; **руками не редактировать**.
- `migrations/*.sql` — нумерованные миграции (последняя: 012).
- `index.html` + `js/timeline*.js`, `js/date-search.js`, `js/date-format.js` — публичная страница.
- `admin.html` + `js/admin.js`, `css/admin.css` — админка (события, черновики, справочники, пользователи, аудит).

## Правила изменений

1. **Схема БД** — только новой миграцией `migrations/NNN_*.sql` (следующий номер по порядку), никогда правкой старых. Сервер применяет их при старте и пишет в `schema_migrations`.
2. **Фильтр публичности** продублирован в двух местах: `is_event_public_ready()` в `server.py` и `isEventPublicReady` в `js/admin.js`. Меняешь правило — меняй **оба** синхронно.
3. Любая запись в `events`/`sources` через API должна проходить `clean_event()` / `clean_reference_item()` и попадать в `audit_log` (исключение: правки черновиков `draft_*` не аудируются — это осознанное решение).
4. После изменения событий сервер сам пересобирает публичные JSON — не добавляй обходных путей записи в БД мимо API.
5. `scripts/convert_csv_to_timelinejson.py` — **устаревший, не использовать**; актуальный путь пересборки только через `server.py`.
6. Форматирование дат — только через общий `js/date-format.js` (`ArchiveDateFormat`), без дублирования логики точности/≈/диапазонов.

## Известные грабли (не воспроизводить)

- **Гонка инициализации TimelineJS:** `window.bindTimelineLayout(timeline)` должен быть вызван **до** `new TL.Timeline(...)`, иначе событие `loaded` может пройти до подписки (особенно при кэшированном JSON). Поэтому `js/timeline-layout.js` использует retry-поллинг и `ResizeObserver` — не «упрощай» это до однократного вызова. Общий класс багов «отрисовка раньше готовности» описан в `open-spec-canvas-timeline.md`.
- Порядок скриптов в `index.html`: `js/vendor/timeline.js` → `js/timeline-layout.js` → создание `TL.Timeline`.
- TimelineJS работает со статическим JSON, а не с БД напрямую — это намеренно.

## Рабочий цикл

1. Прочитай релевантный раздел `open-spec.md` перед правкой.
2. Внеси изменение (код + миграция при необходимости).
3. Проверь: `python server.py`, миграции применились, JSON пересобрался, админка и публичная страница работают, `logs/server.log` без ошибок.
4. Обнови спецификацию и сделай осмысленный коммит (проект под git).

## Чего не делать

- Не редактировать `timeline_data.json` / `scale_data.json` вручную.
- Не менять старые миграции.
- Не разносить правило публичности между сервером и админкой.
- Не менять код без синхронного обновления `open-spec.md`.
