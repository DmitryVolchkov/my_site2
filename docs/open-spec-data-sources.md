# Open Spec: Получение данных из внешних источников

## Связь с основной спецификацией

Этот документ описывает **сбор, верификацию и импорт** исторических данных из интернет-источников в локальную БД архива.

Общая архитектура проекта, схема БД, API, админка и правила публикации — в [open-spec.md](open-spec.md).

Если появляются скрипты импорта, новые поля или API для внешних данных, обновлять **оба** документа.

## Цель

1. Зафиксировать **реестр допустимых источников** и уровни доверия к ним.
2. Описать **единый конвейер**: поиск → извлечение → сопоставление с полями БД → верификация → сохранение через админку или API.
3. Разделить **первичные архивные документы** (подтверждённые) и **вторичные** материалы (энциклопедии, статьи).
4. Дать **эталонный пример** разбора даты **1 сентября 1939 г.** — начала Второй мировой войны в Европе (германское вторжение в Польшу).

## Термины

| Термин | Значение |
| --- | --- |
| **Первоисточник** | Оцифрованный архивный документ с шифром хранения (фонд, опись, дело, лист). |
| **Вторичный источник** | Энциклопедия, учебник, журналистская статья без архивного шифра. |
| **Подтверждённое событие** | Запись в `events` с `verification_status = verified` и хотя бы одним связанным источником с заполненным `evidence_quote`. |
| **ВОВ** | Великая Отечественная война (1941–1945). |
| **ВМВ** | Вторая мировая война (1939–1945); для событий до 22.06.1941 релевантны источники по **предвоенному периоду и началу ВМВ**, а не порталы, ориентированные только на ВОВ. |
| **Период боевых действий** | Для ВОВ — с 22.06.1941 (для СССР) и далее; для остальных участников ВМВ — с даты их вступления в войну. Даты этого периода требуют обязательного шага поиска данных о потерях (Принцип 7). |

---

## Принципы

1. **Один факт — один или несколько первоисточников.** В `sources` указывается URL электронной копии; в `evidence_quote` — цитата или реквизиты документа.
2. **Дата события ≠ дата документа.** Документ может быть датирован 2–3 сентября, но описывать событие 1 сентября; в `events` ставится дата **события**, в `sources.source_date` — дата **документа**.
3. **Публикация на сайте** — только при `status = published` и `verification_status = verified` (см. `is_event_public_ready()` в [open-spec.md](open-spec.md)).
4. **Автоматический импорт** не заменяет редакторскую проверку на первом этапе; скрипт готовит черновик (`status = draft`, `verification_status = unverified`).
5. **robots.txt, лицензии, rate limit** — перед массовым сбором проверять правила каждого сайта; не нагружать государственные порталы частыми запросами.
6. **Не смешивать уровни доверия:** вторичный текст может идти в `summary`, но не может быть единственным основанием для `verified`.
7. **Потери за период боевых действий.** Если целевая дата приходится на период боевых действий (для ВОВ — с 22.06.1941), конвейер обязательно включает отдельный шаг поиска данных о потерях (безвозвратных и санитарных) на эту дату — см. [«Потери и погибшие»](#потери-и-погибшие-обязательный-шаг-для-дат-периода-боевых-действий) ниже. Если подневных данных не найдено, это фиксируется явно как ограничение в `text` события, а не подменяется агрегированными периодами без пометки уровня источника (см. Принцип 6).

## Конвейер получения данных

```text
┌──────────────────────────────────────────────────────────────────┐
│ 1. Постановка задачи                                              │
│    Целевая дата / тема / географический фильтр                    │
└────────────────────────────┬─────────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│ 2. Выбор источников (таблица «Применимость по периоду»)           │
└────────────────────────────┬─────────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│ 3. Поиск на портале (ручной или scripts/fetch_sources.py — TBD)   │
│    Фиксация: URL, архивный шифр, заголовок, дата документа       │
└────────────────────────────┬─────────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│ 4. Извлечение                                                    │
│    headline, summary, text, evidence_quote, media_url            │
└────────────────────────────┬─────────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│ 5. Черновик в админке (вкладка «Черновики»)                       │
│    Лист = запрос/дата → строка на событие → источники по клику   │
│    POST /api/drafts/batches → /events → /sources (без строгой    │
│    валидации; редактор правит прямо в таблице перед импортом)    │
└────────────────────────────┬─────────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│ 6. Импорт → clean_event() / clean_reference_item()                │
│    POST /api/drafts/events/:id/import: start_year/month/day,     │
│    country_name, domain — те же правила, что и у обычной формы   │
└────────────────────────────┬─────────────────────────────────────┘
                             ▼
┌──────────────────────────────────────────────────────────────────┐
│ 7. Сохранение в основной БД                                      │
│    events (status = draft) + sources, связка event_sources       │
│    Редактор: verification_status → verified → published          │
└──────────────────────────────────────────────────────────────────┘
```

Шаги 5–6 реализованы через SQLite-таблицы `draft_batches`/`draft_events`/`draft_sources` и вкладку «Черновики» в админке — см. [open-spec.md](open-spec.md#draft_batches) для схемы и API. Это заменяет собой изначально запланированный файловый формат (см. ниже).

## Потери и погибшие (обязательный шаг для дат периода боевых действий)

Если целевая дата запроса попадает в период боевых действий (для ВОВ — с 22.06.1941; для ВМВ в целом — соответствующие периоды кампаний), конвейер (шаг 3–4, «Поиск» / «Извлечение») обязательно включает отдельный подшаг: поиск данных о потерях (безвозвратных и санитарных) конкретно на эту дату, а не только политических/дипломатических/военно-оперативных событий.

### Порядок поиска

1. **Первоисточники частей/соединений** на `prlib-vov` — донесения о потерях, оперативные и разведывательные сводки штабов, сводки политотделов и погранвойск НКВД за нужную дату (запрос вида `site:prlib.ru item донесение о потерях <дата/дивизия/фронт>`). Это основной источник для `verified`.
2. **`pamyat-naroda`** — применим с 1941 г.; используется для проверки/уточнения по конкретной части, соединению или донесению о потерях, а не как источник готовой цифры «погибло N человек в такую-то дату» — портал не агрегирует потери по датам, только по частям/персоналиям/донесениям.
3. **`sovdoc-war`, `victory`** — фото-, кино-, документальные комплексы с 1941 г.; проверять на наличие сводок потерь по нужному периоду.
4. **Историографические своды уровня B** (например, межведомственная комиссия Г. Ф. Кривошеева «Гриф секретности снят», публикуется в т. ч. на `history-ru`) дают только периодные/фронтовые агрегаты (например, «Западный фронт, 22.06–09.07.1941»), а не подневные цифры. Их можно приводить в `text` события как контекст с явной пометкой уровня источника и периода, который они покрывают — **не приписывать эту цифру конкретной дате события**.

### Если подневных данных не найдено

Для первых дней ВОВ это частая ситуация: штабы физически не успевали подсчитать потери за 22–23 июня 1941 года — донесения о потерях собирались частями и обобщались фронтами с задержкой в дни и недели. В этом случае:

- в `text` события явно фиксируется, что подневных данных о потерях на эту дату не найдено (см. пример по 22–23.06.1941 ниже);
- запрещено подставлять периодный/фронтовой агрегат вместо отсутствующей подневной цифры без явной оговорки, что это агрегат за более широкий период;
- `verification_status` события о потерях не может быть `verified` только на основании историографического свода уровня B (Принцип 6).

### Пример (эталон): потери на 22–23.06.1941

Разбор по аналогии с примером 1 сентября 1939 г. ниже: на 22–23.06.1941 найдены два первоисточника уровня A с прямым отношением к первому дню боёв (сводка погранвойск НКВД Молдавского округа, РГВА. Ф. 32880. Оп. 5. Д. 230. Л. 1; оперативная сводка № 01 Генштаба РККА, ЦА МО РФ. Ф. 228. Оп. 701. Д. 84. Л. 1-4) — ни один из них не содержит сводного числа погибших за день. В `text` обоих событий как контекст приведена периодная цифра комиссии Кривошеева (Западный фронт, 22.06–09.07.1941: 341 021 безвозвратных + 76 717 санитарных = 417 729) с явной оговоркой, что это агрегат за 18 дней и несколько фронтов, а не за 22 или 23 июня отдельно.

## Сопоставление с полями БД

### Таблица `events`

| Поле | Откуда брать | Правило |
| --- | --- | --- |
| `start_year`, `start_month`, `start_day` | Дата **события** | Если известен только месяц — `start_date_precision = month` |
| `start_date_precision` | Точность | `day` / `month` / `year` / `approximate` |
| `headline` | Редакционный заголовок портала или краткая формулировка редактора | Нейтральный стиль, без оценочных клише |
| `summary` | 1–3 предложения по документу | Не дублировать `text` |
| `text` | Развёрнутое описание + контекст | Ссылки на источники в конце |
| `event_type` | Классификация | `war`, `diplomacy`, `politics` и т.д. |
| `scale` | Масштаб | `international`, `national`, `regional` |
| `domain` | Сфера | `military`, `politics`, `diplomacy` |
| `country_name` | География события | «Польша», «Германия», «СССР» |
| `verification_status` | После проверки | `unverified` → `verified` |
| `status` | Жизненный цикл | `draft` → `published` |

### Таблица `sources`

| Поле | Откуда брать | Пример |
| --- | --- | --- |
| `title` | Название документа на портале | «Телеграмма Н. И. Шаронова…» |
| `url` | Постоянная ссылка на электронную копию | `https://docs.historyrussia.org/ru/nodes/147507-…` |
| `type` | Тип | `archive`, `article`, `government` |
| `author` | Автор / орган | «Н. И. Шаронов», «ТАСС» |
| `source_date` | Дата **документа** | `1939-09-01` |
| `citation` | Библиографическое описание | «АВП СССР, ф. 059, оп. 1, п. 296, д. 2046, л. 268» |
| `reliability_score` | Оценка | `9–10` для уровня A с шифром |
| `evidence_quote` | Цитата или выдержка | Фрагмент текста или суть донесения |

### Связи

- `event_sources` — каждое событие связано со всеми использованными первоисточниками.
- `event_tags` — тематические метки: `#ВтораяМировая`, `#1939`, `#Польша`.
- `event_groups` — полоса таймлайна (например, «Международная политика»).

## Черновик импорта (реализовано через админку)

Изначально здесь планировался промежуточный JSON-файл `data/import_drafts/<date>_<slug>.json` и отдельный скрипт `scripts/import_from_draft.py`. Вместо этого промежуточный этап реализован как вкладка **«Черновики»** в админке поверх новых SQLite-таблиц — без файлов и без отдельного скрипта:

- «Лист» (`draft_batches`) соответствует бывшему `meta.target_date` — один лист на дату/тему запроса.
- Строка листа (`draft_events`) соответствует одному объекту из бывшего массива `events`: `headline`, `start_year`/`start_month`/`start_day`, `summary`, `text`, классификация.
- Источники строки (`draft_sources`, разворачиваются по клику) соответствуют бывшему вложенному массиву `sources`: `url`, `citation`, `evidence_quote` (поле `registry_id` из реестра источников выше сейчас фиксируется вручную в `title`/`type`, отдельного поля под ID реестра пока нет).
- Вместо отдельного скрипта импорта — кнопка «Импортировать» на строке, которая дёргает `POST /api/drafts/events/:id/import` и переиспользует **тот же** `clean_event()`/`clean_reference_item()`, что и обычная форма события в «Наполнении» (см. [open-spec.md](open-spec.md) — раздел API, `POST /api/drafts/events/:id/import`).

Так собранные данные видно и можно поправить в таблице до того, как они попадут в `events`/`sources`, без ручного редактирования JSON и без риска разойтись с правилами валидации основной формы.

## Пример анализа: 1 сентября 1939 года

### Контекст даты

**1 сентября 1939 г.** — день **германского вторжения в Польбу**; в мировой historiography это принято считать **началом Второй мировой войны в Европе**. Для СССР это ещё **предвоенный период** (до 22.06.1941), поэтому:

- порталы **ВОВ** (`pamyat-naroda`, `victory`, `sovdoc-war`) **не содержат** боевых документов на эту дату;
- релевантны **`prlib-vov`**, **`nakanune`**, **`ebid`**, материалы **АВП СССР / РГВА** через Президентскую библиотеку и ЭБИД.

---

### Шаг 1. Поиск по реестру

| Источник | Запрос / раздел | Результат |
| --- | --- | --- |
| `prlib-vov` | Раздел «1939 год → Сентябрь» | Документы с датой 1 сентября 1939 |
| `nakanune` | Хронология 1939 | Документы предвоенного периода |
| `ebid` | Поиск: «1 сентября 1939» | Телеграммы, дипломатическая переписка |
| `pamyat-naroda` | — | **Не применим** (ВОВ с 1941) |

---

### Шаг 2. Найденные первоисточники (эталон)

#### Документ A — начало германо-польской войны (международный контекст)

| Реквизит | Значение |
| --- | --- |
| **Портал** | `prlib-vov` |
| **URL** | https://www.prlib.ru/item/1309187 |
| **Заголовок** | Из служебного выпуска № 221/с иностранной информации ТАСС — о выступлении рейхсканцлера Германии А. Гитлера в рейхстаге в связи с началом германо-польской войны |
| **Дата документа** | 1 сентября 1939 г. |

### Шаг 5. Чеклист верификации редактора

- [ ] Дата события подтверждена **первоисточником**, а не только энциклопедией.
- [ ] Заполнены `citation` (архивный шифр) и `url` (устойчивая ссылка).
- [ ] `evidence_quote` содержит проверяемый фрагмент или точное пересказанное содержание документа.
- [ ] `reliability_score` ≥ 8 для уровня A.
- [ ] Событие не дублирует существующую запись в БД (проверка по дате + `headline`).
- [ ] Для 1939–1940 не использованы как единственные источники порталы только ВОВ.
- [ ] Если дата входит в период боевых действий — выполнен поиск данных о потерях на эту дату (Принцип 7, раздел «Потери и погибшие»); если подневных данных нет — это явно зафиксировано в `text`, а не подменено периодным агрегатом без пометки.
- [ ] `status = published` только после финальной вычитки.

## Ограничения и риски

| Риск | Митигация |
| --- | --- |
| OCR-ошибки в ЭБИД | Сверка со сканом на prlib или запрос в читальный зал |
| Платный доступ к сканам (ГИС УИАД) | Метаданные и шифр — бесплатно; полный текст — по политике проекта |

| Нормативная терминология | Единый стиль заголовков; спорные формулировки — `verification_status = disputed` |

## Критерии приёмки спецификации

1. Редактор по этому документу может вручную создать **два проверенных события** на 01.09.1939 с первоисточниками.
2. Ясно, какие порталы **не применимы** к датам до 22.06.1941.
3. Каждое поле `events` и `sources` имеет правило происхождения из внешнего источника.
4. Конвейер совместим с существующим API и `clean_event()` без изменения схемы `events`/`sources` (черновики хранятся в отдельных таблицах `draft_*`, добавленных отдельной миграцией).
5. Для любой даты периода боевых действий конвейер явно предусматривает поиск данных о потерях (Принцип 7, раздел «Потери и погибшие»), включая случай, когда подневных данных не найдено и это зафиксировано как ограничение, а не подменено периодным агрегатом.
