# Open Spec: Архив истории

## Цель

Проект хранит исторические события, показывает их на интерактивной хронологии и даёт админке возможность добавлять, редактировать и удалять записи.

Спецификация фиксирует договорённости по данным, API и поведению интерфейса. Если код меняется, этот документ нужно обновлять вместе с ним.

Пошаговая инструкция для ручного ввода данных через админку: [manual-entry/README.md](manual-entry/README.md).

Сбор данных из внешних интернет-источников (реестр, конвейер, пример 01.09.1939): [open-spec-data-sources.md](open-spec-data-sources.md).

## Текущая Архитектура

- `server.py` запускает локальный HTTP-сервер и API.
- `data/archive.sqlite3` хранит основную базу данных SQLite.
- `data/events.csv` используется как источник первичного импорта при создании пустой БД.
- `data/timeline_data.json` и `data/scale_data.json` автоматически пересобираются из SQLite для TimelineJS и шкалы.
- `migrations/*.sql` хранит историю изменений схемы БД.
- `index.html` показывает хронологию из JSON-файлов, собранных сервером из БД.
- `admin.html` управляет событиями, черновиками, пользователями, справочниками, аудитом и структурой БД через API.
- `js/admin.js` отвечает за интерфейс админки, включая модальный предпросмотр публичной карточки события.
- `js/date-format.js` (`ArchiveDateFormat`) — общая логика форматирования дат: точность, ≈, диапазон начало—окончание; используется в админке (подсказка, предпросмотр) и на главной странице.
- `js/date-search.js` отвечает за поиск по дате и отображение выбранного факта на главной странице.
- `js/timeline-layout.js` подгоняет DOM/CSS TimelineJS под вёрстку сайта (menubar слева, фиксированная высота, скрытие превью медиа на флажках) без замены узлов слайдера; из-за гонки инициализации TimelineJS (см. ниже) использует поллинг с ретраями и `ResizeObserver`, а не однократный вызов после события `loaded`.
- `scripts/convert_csv_to_timelinejson.py` — устаревший автономный скрипт; **не использовать** для пересборки JSON. Актуальный путь — `server.py` → `events_to_timeline()` / `write_timeline_files()`.

## Поток Данных

Основной источник данных: `data/archive.sqlite3`.

Поток для публичной страницы:

1. Сервер запускается командой `python server.py`.
2. Сервер применяет миграции.
3. Если БД пустая, сервер импортирует начальные события из `data/events.csv`.
4. Сервер пересобирает `data/timeline_data.json` и `data/scale_data.json` из SQLite.
5. В публичные JSON-файлы попадают только события с `status = published` и `verification_status = verified` (фильтр `is_event_public_ready()` в `events_to_timeline()` и `events_to_scale()`; то же правило в админке — `isEventPublicReady` в `js/admin.js`).
6. `index.html` передаёт `data/timeline_data.json` в TimelineJS.
7. Поиск по дате использует тот же `data/timeline_data.json`.

**Порядок загрузки скриптов важен:** `js/timeline-layout.js` должен быть загружен и `window.bindTimelineLayout(timeline)` вызван **до** первой отрисовки `new TL.Timeline(...)`, а не после. Если конструктор `TL.Timeline` создаётся раньше, чем подписан обработчик `timeline.on('loaded', …)`, событие `loaded` может произойти до подписки (особенно при быстрой загрузке `timeline_data.json` из кэша) — таймлайн останется без обвязки `timeline-layout.js` до первого resize. Текущий `index.html` грузит `js/vendor/timeline.js` и `js/timeline-layout.js` последовательно и только затем создаёт `TL.Timeline`.

Это частный случай более общего класса гонок инициализации («отрисовка раньше, чем контейнер/событие готовы»), из-за которого `js/timeline-layout.js` держит retry-поллинг и `ResizeObserver` вместо одноразового вызова после `loaded` — см. «Уроки для нового рендера» в [open-spec-canvas-timeline.md](open-spec-canvas-timeline.md), где тот же класс багов заложен в требования к будущему Canvas-рендеру, чтобы не воспроизводить его заново.

Поток для админки:

1. Пользователь входит через `POST /api/auth/login`; сессия хранится в cookie `archive_session`.
2. Админка загружает события через `GET /api/events`, справочники через `/api/sources`, `/api/media`, `/api/tags`, `/api/groups`.
3. Сохранение события идёт через `POST /api/events`.
4. Удаление события идёт через `DELETE /api/events/:id`.
5. После сохранения или удаления сервер обновляет SQLite и пишет запись в `audit_log`.
6. После изменения событий сервер автоматически пересобирает JSON-файлы для публичной страницы.
7. Администратор может просматривать последние изменения через `GET /api/audit`.

Важно: TimelineJS стабильно работает со статическим JSON-файлом `data/timeline_data.json`. Поэтому БД не отдаётся в TimelineJS напрямую, а сначала преобразуется в совместимый JSON.

Поток для черновиков (промежуточный этап перед «Наполнением», см. [open-spec-data-sources.md](open-spec-data-sources.md)):

1. В разделе «Черновики» редактор создаёт «лист» (`draft_batches`) — обычно один лист на дату/запрос.
2. Внутри листа заполняется таблица строк-черновиков (`draft_events`) с инлайн-редактируемыми ячейками; изменения автосохраняются через `POST /api/drafts/events` по потере фокуса поля (без записи в `audit_log` — черновиковые правки не аудируются).
3. У каждой строки может быть несколько источников (`draft_sources`), которые разворачиваются по клику и тоже автосохраняются через `POST /api/drafts/sources`.
4. Кнопка «Импортировать» на строке вызывает `POST /api/drafts/events/:id/import`: сервер проверяет `headline` и `start_year`, создаёт запись в `events` (через тот же `clean_event()`, что и обычная форма, с явным `status = draft`) и переносит источники в `sources` (через `clean_reference_item()`), связывая их через `event_sources`.
5. После импорта строка помечается `import_status = imported` и остаётся в листе для истории; дальнейшая проверка и публикация происходят как обычно — в разделе «Наполнение».
6. Удаление листа каскадно удаляет его строки и источники черновика, но не трогает уже импортированные записи в `events`/`sources`.

### Связи Событий

- `event_sources` связывает события с источниками.
- `event_media` связывает события с медиа.
- `event_tags` связывает события с тегами.
- `event_groups` связывает события с группами.

`GET /api/events` отдаёт для каждого события массивы `source_ids`, `media_ids`, `tag_ids`, `group_ids`, а также развёрнутые `source_items`, `media_items`, `tag_items`, `group_items`.

`POST /api/events` принимает эти массивы и сохраняет связи в `event_sources`, `event_media`, `event_tags`, `event_groups`.

`data/timeline_data.json` включает связанные записи в служебных полях `_sources`, `_media_items`, `_tag_items`, `_group_items`, а также атрибуты фазы 1 в полях `_hashtag`, `_summary`, `_event_type`, `_scale`, `_domain`, `_category`, `_subcategory`, `_country_name`, `_region`, `_city`, `_verification_status`, `_start_date_precision`, `_end_date_precision`, `_start_date_approximate`, `_end_date_approximate`, `_related_events`.

Публичная карточка события показывает связанные источники, медиа и теги под описанием события.

Поля `media_url`, `media_caption`, `media_credit` и строковое поле `tags` в `events` пока сохраняются для совместимости с TimelineJS.


При запуске `server.py` сервер:

1. Создаёт служебную таблицу `schema_migrations`, если её ещё нет.
2. Смотрит, какие миграции уже применены.
3. Берёт все файлы `*.sql` из папки `migrations` по порядку имени.
4. Применяет только новые миграции.
5. Записывает применённую миграцию в `schema_migrations`.

Чтобы изменить схему БД, нужно добавлять новый файл, например:

```text
migrations/002_add_users_table.sql
```

Старые миграции после применения лучше не редактировать: они являются историей схемы.

### `POST /api/events/bulk/attach`, `POST /api/events/bulk/verify`, `POST /api/events/bulk/publish`

Массовые операции над выбранными событиями из раздела «База данных» (чекбоксы на строках). **Требуют роль `admin`** — строже обычного редактирования одного события (`admin` или `editor`), т.к. затрагивают сразу много записей.

Тело запроса — список `id`:

```json
{ "ids": ["ev-0001", "ev-0002"] }
```

Для `bulk/attach` дополнительно передаётся целевое событие:

```json
{ "ids": ["ev-0001", "ev-0002"], "attached_to": "ev-0009" }
```

Каждый `id` обрабатывается независимо: сервер читает существующую строку события **вместе с текущими `source_ids`/`media_ids`/`tag_ids`/`group_ids`** (`get_event_for_update()`), точечно меняет одно поле (`attached_to`, `verification_status = "verified"` или `status = "published"`), прогоняет через тот же `clean_event()` и (для `attach`) `validate_event_attachment()`, что и обычное сохранение формы, и вызывает `upsert_event()`. Ошибка по одному `id` (например, дата не совпадает с целевым событием при `attach`, или `publish` без `verified`) не прерывает обработку остальных. Одна запись в `audit_log` на каждое успешно изменённое событие, один вызов пересборки JSON-файлов на весь запрос.

Ответ:

```json
{
  "results": [
    { "id": "ev-0001", "ok": true },
    { "id": "ev-0002", "ok": false, "error": "Опубликовать можно только проверенные события: установите «Проверка фактов» = verified." }
  ]
}
```

Рабочий сценарий публикации нескольких событий: сначала `bulk/verify`, затем `bulk/publish`.

## Правила Данных

- `id` должен быть стабильным: после публикации его нельзя менять без причины.
- `start_year` обязателен для каждого события.
- `headline` обязателен для каждого события.
- Если известен только год, `start_month` и `start_day` остаются пустыми.
- Для длительных событий используются поля `end_year`, `end_month`, `end_day`.
- Теги можно писать через точку с запятой: `СССР;Праздники`.
- Допустимые статусы публикации: `draft`, `review`, `published`.
- Допустимые статусы проверки: `verified`, `needs_review`, `disputed`, `unconfirmed`.
- **Связь статусов:** на главную попадают только события с `status = published` и `verification_status = verified` — при сохранении через API, в админке, предпросмотре и при экспорте JSON (`is_event_public_ready()` / `isEventPublicReady`).
- При `status = published` и пустой проверке сервер автоматически ставит `verification_status = verified`.
- При `status = published` и проверке `needs_review`, `disputed` или `unconfirmed` сохранение через API отклоняется.
- При `status = review` без проверки сервер ставит `verification_status = needs_review`.
- При `status = draft` без проверки сервер ставит `verification_status = unconfirmed`.
- `hashtag` — короткая метка без пробелов; `#` при вводе необязателен, в интерфейсе показывается с `#`.
- Допустимые значения `start_date_precision` и `end_date_precision`: `day`, `month`, `year`, `approximate`; пустое значение = **Авто**.
- **Авто (начало):** день+месяц+год → `day`; месяц+год → `month`; только год → `year`.
- **Авто (окончание):** по тем же правилам для полей окончания.
- Если задан год окончания, на карточке показывается диапазон `начало — окончание` с отдельной точностью для каждой стороны.
- **≈ Приблизительная** добавляет префикс `≈` к соответствующей дате.
- Точность `approximate` автоматически включает чекбокс ≈ на той же стороне.
- Публичная карточка и предпросмотр форматируют дату через `js/date-format.js` (`ArchiveDateFormat.formatEventDate`).
- Допустимые значения `scale`: `local`, `national`, `regional`, `international`.
- **Группа по умолчанию:** если у **нового** события (при первом сохранении, `INSERT`) не указан ни один `group_ids`, `upsert_event()` в `server.py` автоматически привязывает его к группе «История» (`grp-0001`) и заполняет строковое поле `group` её названием. Действует для любого пути создания события — формы «Наполнение», импорта черновика, CSV-импорта. При обновлении **существующего** события группу принудительно не восстанавливает — можно осознанно оставить событие без группы.
- Отображение группы в таблицах админки (`renderEventsTable()` в `js/admin.js`) берёт название из `group_items` (актуальная связь `event_groups`), и только если связей нет — использует устаревшее строковое поле `group`. Оба поля обязаны быть согласованы при сохранении события (сервер синхронизирует `group` и `group_ids` для новых событий; при ручных правках БД в обход API синхронизировать нужно вручную).
- `hashtag` события должен быть уникальным, если задан.
- `related_events` хранит коды событий через `;` или `,`.
- В публичный таймлайн и шкалу попадают только события с `published` + `verified`.
- Первичный импорт из `data/events.csv` (`import_csv`) проходит `clean_event()` — те же правила дат, статусов и хештега, что и API. Невалидные строки пропускаются с сообщением в консоль.
- Заголовки `data/events.csv` совпадают с `EVENT_COLUMNS` (36 полей). Инструкция: [manual-entry/README.md](manual-entry/README.md).
- Допустимые роли пользователей: `admin`, `editor`, `viewer`.
- Пароли пользователей не хранятся в открытом виде.
- Изменять события через API могут только `admin` и `editor`.
- Управлять пользователями может только `admin`.
- Создание, изменение и удаление событий, пользователей, источников, медиа и тегов фиксируется в `audit_log`.
- CSV больше не считается основной БД, он нужен только для импорта или резервной копии.
- `data/timeline_data.json` и `data/scale_data.json` не редактируются вручную: они генерируются из SQLite.
- Черновики (`draft_batches`/`draft_events`/`draft_sources`) не проходят строгую валидацию `clean_event()` — обязательные поля проверяются только при импорте (`headline`, `start_year`).
- Импорт черновика всегда создаёт событие со `status = draft`; при отсутствии этого значения `clean_event()` подставил бы `published` — поэтому обработчик импорта передаёт `status` явно.
- Автосохранение черновиков (по потере фокуса поля) не пишет в `audit_log`; запись аудита создаётся один раз — в момент импорта строки в `events`.
- `attached_to` (составные события): основное событие, указанное в `attached_to`, должно существовать и само не быть вторичным (без цепочек — только один уровень «основное → вторичное»).
- Событие, у которого уже есть свои вторичные (кто-то ссылается на него через `attached_to`), нельзя привязать к другому основному — сначала нужно отвязать его собственные вторичные события.
- Дата вторичного события (`start_year`/`start_month`/`start_day`) должна точно совпадать с датой основного — иначе сервер отклоняет сохранение (`validate_event_attachment()` в `server.py`); при разных датах это должно быть отдельное событие без `attached_to`.
- Вторичное событие нельзя опубликовать (`published` + `verified`), пока основное само не `published` + `verified`; обратное разрешено — основное можно публиковать без готовых вторичных.

## Админка

Разделы админки:

- **Наполнение** — события.
- **Черновики** — промежуточный этап сбора данных до попадания в «Наполнение», для `admin` и `editor`; чекбоксы на строках + «Импортировать выбранные» доступны без доп. ограничения роли.
- **Пользователи** — только для `admin`.
- **Источники**, **Медиа**, **Теги**, **Группы** — справочники для `admin` и `editor`.
- **Аудит** — последние изменения, только для `admin`.
- **База данных** — список всех событий с CSV-экспортом, для `admin` и `editor`; чекбоксы на строках и bulk-тулбар (назначение «Привязки», «Пометить проверенным», «Опубликовать») видны и работают **только для `admin`**.
- **Структура БД** — просмотр схемы SQLite, только для `admin`.

Админка должна:

- Загружать события из `/api/events`.
- Сохранять событие через `POST /api/events`.
- Удалять событие через `DELETE /api/events/:id`.
- Загружать листы и строки черновиков из `/api/drafts`, сохранять их через `POST /api/drafts/batches|events|sources` и импортировать строку в «Наполнение» через `POST /api/drafts/events/:id/import`.
- Для `admin`: массово назначать `attached_to`, `verification_status = verified` и `status = published` выбранным событиям в «База данных» через `POST /api/events/bulk/attach|verify|publish`.
- Позволять выбрать статус публикации: черновик, на проверке или опубликовано.
- Позволять задавать атрибуты фазы 1: `hashtag`, `summary`, классификацию, географию, точность даты, статус проверки и связанные события.
- Позволять выбрать связанные источники, медиа, теги и группы из справочников в форме события.
- Позволять указывать `evidence_quote` в форме источника.
- Давать поиск, счётчик выбранного и быстрый переход к созданию записи при выборе справочников в форме события.
- Показывать модальный предпросмотр публичной карточки события поверх админки.
- Давать возможность скачать CSV-копию текущих данных.
- Показывать структуру БД через раздел “База данных”.
- Показывать последние записи аудита в разделе “Аудит”.
- После сохранения или удаления события публичный таймлайн должен получать обновлённые JSON-файлы.
- Выполнять вход через серверный endpoint, а не через `localStorage`.
- Управлять пользователями через SQLite и серверный API.
- Управлять источниками, медиа, тегами и группами через отдельные разделы админки.

### Форма события (вёрстка)

Компактная верхняя строка `.admin-event-meta-bar` (на всю ширину):

- **Код** (`event-id`, только чтение, автогенерация);
- **Публикация** (`event-status`: черновик / на проверке / опубликовано);
- **Проверка** (`event-verification-status`);
- **Важн.** (`event-importance`, 1–10);
- **Хештег** (`event-hashtag`).

Подсказка `#event-status-hint` объясняет, попадёт ли событие на главную.

Блок дат `.admin-date-range` — две колонки **ДАТА НАЧАЛА** и **ДАТА ОКОНЧАНИЯ**:

| Строка | Содержимое | Размеры полей |
| --- | --- | --- |
| 1-я (`.admin-date-values`) | День, Месяц, Год | 55px, 75px, 100px |
| 2-я (`.admin-date-options`) | ≈ Приблизительная, селект точности | чекбокс + селект 92px |

Подсказка `#event-date-hint` показывает итоговую дату по правилам `ArchiveDateFormat`.

Связи со справочниками выбираются через модальные меню **+ Источник / + Медиа / + Тег / + Группа** (Новый / База / Папка для медиа). Строковое поле `tags` в форме не редактируется вручную — собирается из выбранных тегов. Поля `media_url`, `media_caption`, `media_credit` скрыты в форме; основное медиа задаётся через справочник или загрузку файла.

Блок «Объединение на шкале» — поля `attached_to`, `attachment_order`, `composite_headline`, `use_composite_headline` (см. «Составные события» ниже); подсказка `#event-attachment-hint` реактивно проверяет совпадение даты с выбранным основным событием ещё до отправки формы.

Стили формы: `css/admin.css` (классы `admin-date-*`, `admin-event-meta-bar`).

### Предпросмотр события

Кнопка **«Предпросмотр карточки»** в разделе “Наполнение” открывает модальное окно поверх админки.

Предпросмотр повторяет публичный блок факта с главной страницы:

- заголовок и хештег;
- дату (с точностью и ≈), место, группу и краткие метаданные (`scale`, `event_type`, `domain`);
- краткое (`summary`) и полное описание;
- основное медиа события (`media_url` → блок `media` в JSON);
- связанные источники (с `evidence_quote`), медиа и теги из выбранных справочников.

Поведение:

- пока окно открыто, содержимое обновляется при изменении полей формы;
- предупреждение, если событие **не готово** к главной: `draft`, `review`, или `published` без `verified`;
- закрытие: кнопка **«Закрыть»**, клик по затемнённому фону или клавиша `Esc`.

### Черновики (Excel-подобная сетка)

Раздел «Черновики» организован как книга Excel: «листы» (`draft_batches`) — обычно один лист на дату или тему запроса, переключаются кликом в вертикальном списке (`#draft-sheets-list`) слева от таблицы, а не пилюлями над ней (было заменено — список масштабируется на много листов лучше, чем горизонтальная полоса).

Над списком листов — фильтры по году/месяцу/дню (`#draft-filter-year|month|day` + кнопка «Сбросить»): дата листа (`target_date`) хранится свободным текстом (может быть диапазоном вроде `1939-09-02..1941-06-22`), фильтры парсят **ведущую** дату `YYYY-MM-DD` через `parseDraftBatchDate()`; список года собирается динамически из реально существующих дат, месяц/день — статичные 01–12/01–31. Лист с нераспознаваемой датой не участвует в фильтрации, но остаётся виден в списке, пока фильтры не заданы. Активный лист не снимается фильтром, даже если перестал ему соответствовать — фильтр помогает найти лист, а не прячет уже открытый. Каждый пункт списка показывает название, дату и счётчик «импортировано/всего строк».

Внутри листа — редактируемая таблица (`.draft-grid`) без отдельной формы сверху: каждая ячейка — инлайн `<input>`/`<select>`, изменения автосохраняются по потере фокуса поля (debounce ~400мс) через `POST /api/drafts/events`. Источники события (их может быть несколько) не выводятся отдельными столбцами — строка разворачивается по клику на счётчик, показывая под-таблицу источников и два textarea (`summary`, `text`).

Кнопка **«Импортировать»** доступна, пока строка не импортирована; после успешного импорта строка помечается бейджем «Импортировано», поля блокируются на чтение, а кнопка исчезает — редактирование продолжается уже в «Наполнении» на созданном событии.

Создание листа — через инлайн-форму (название + целевая дата), а не через `prompt()`: нативные диалоги не соответствуют стилю остальной админки и не нужны для одного текстового поля.

**Массовый импорт:** первый столбец таблицы — чекбокс строки (у уже импортированных строк — отключён); заголовок таблицы содержит чекбокс «выбрать все». Как только выбрана хотя бы одна строка, над таблицей появляется тулбар со счётчиком и кнопкой «Импортировать выбранные» — она последовательно вызывает `POST /api/drafts/events/:id/import` для каждой отмеченной строки и показывает сводку («Импортировано N из M», с текстом ошибки для каждой не прошедшей валидацию строки), не прерываясь на первой ошибке.

### База данных — bulk-действия (только `admin`)

В таблице «База данных» первый столбец — чекбокс строки; виден только пользователю с ролью `admin` (для `editor` столбец и весь bulk-тулбар скрыты через `canManageUsers()` на клиенте — сервер дополнительно проверяет роль на каждом `POST /api/events/bulk/*`). Тулбар над таблицей появляется при выборе хотя бы одной строки:

- **«Назначить привязку»** — текстовое поле с `datalist` (список событий без своего `attached_to`, тот же источник данных, что и даталист в форме события) + кнопка «Применить» → `POST /api/events/bulk/attach`.
- **«Пометить проверенным»** → `POST /api/events/bulk/verify`.
- **«Опубликовать»** → `POST /api/events/bulk/publish`.

Каждая кнопка подтверждается через `confirm()` с числом затронутых событий, после ответа сервера — сводка результатов в `#events-db-status` (сколько успешно, что не удалось и почему), перезагрузка таблицы из `/api/events`, сброс выделения.

### Составные события (объединение на шкале)

Если у события есть вторичные (одно или несколько других событий с `attached_to`, указывающим на него), оно показывается на шкале и в поиске по дате **одной точкой** вместо нескольких — так решается ситуация, когда два самостоятельных события с одной и той же датой иначе конкурировали бы за один и тот же флажок TimelineJS/карточку факта.

Как это устроено:

- `events.attached_to` — ID основного события; если задан, событие вторичное. Один основной может иметь сколько угодно вторичных, но только один уровень вложенности (вторичное не может само быть основным для другого).
- `events.attachment_order` — ручной порядок вторичного события внутри объединённой карточки (без значения — по `id`).
- `events.composite_headline` + `events.use_composite_headline` — задаются на основном событии; если чекбокс включён и текст не пуст, вместо обычного `headline` на шкале и в карточке показывается `composite_headline`.
- `validate_event_attachment()` в `server.py` проверяет при сохранении: основное существует, не является само вторичным, дата совпадает, и вторичное не публикуется раньше основного (см. «Правила Данных»).
- `events_to_timeline()` / `events_to_scale()` группируют вторичные события по основному (`_attached_members_by_primary()`), убирают их из списка отдельных маркеров и добавляют в маркер основного поле `_attached_events` — массив с `headline`, `summary`, `text`, `media`, `source_items`, `media_items`, `tag_items` каждого вторичного. В группу попадают только вторичные, которые сами прошли `is_event_public_ready()` **и** у которых основное тоже прошло — иначе вторичное не показывается ни отдельно, ни в составе объединённой карточки.
- `js/date-search.js` (`renderAttachedMembers()`) рендерит каждый элемент `_attached_events` под-блоком со своим заголовком-подписью внутри «Кратко» и «Описание», и отдельной секцией «Источники/Медиа/Теги» — так каждая часть составного факта сохраняет привязку к своим источникам.
- В админке форма события содержит блок «Объединение на шкале»: поле «Привязано к» (с datalist кандидатов на роль основного, исключая уже вторичные события), «Порядок в объединении» и «Объединяющий заголовок» с чекбоксом «Использовать вместо заголовка». Список событий («База данных») показывает бэйдж привязки (`→ ev-0009` или `+N привязано`).
- Случайное совпадение дат у **не связанных** через `attached_to` событий — отдельный случай: `js/date-search.js` (`findBestEvent()`/`searchByDate()`) показывает их как несколько независимых карточек друг под другом в `#fact-extra-matches`, а не молча теряет все, кроме первой.

### 6. Убрать дублирование legacy-полей в `events`

**Суть:** в таблице `events` одновременно хранятся строковые `tags`, `media_url`, `media_caption`, `media_credit` и нормализованные связи через `event_tags`, `event_media`.

**Что сделать:** после уверенного перехода на справочники — генерировать `tags` и `media` для TimelineJS только в `events_to_timeline()`, не дублируя в БД; либо оставить колонки read-only для обратной совместимости.

**Зачем:** один источник правды, меньше рассинхрона между строковым полем и справочником.

**Риск:** старые записи и CSV-экспорт могут зависеть от этих колонок — миграция потребует скрипта переноса данных.

### 7. Предпросмотр в админке — как на главной

**Суть:** модальный предпросмотр (`js/admin.js`) пока склеивает краткое и полное описание в один абзац; на главной они разделены («Кратко» / «Описание», сворачивание, медиа внизу полного блока).

**Что сделать:** вынести общую разметку/логику карточки в переиспользуемый фрагмент или общие функции рендера; предпросмотр должен совпадать с `index.html`.

**Зачем:** редактор видит на экране то же, что увидит посетитель, без сюрпризов после публикации.

**Где менять:** `admin.html` (разметка модалки), `js/admin.js`, возможно общий модуль рядом с `date-format.js`.
