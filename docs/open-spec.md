# Open Spec: Архив истории

## Цель

Проект хранит исторические события, показывает их на интерактивной хронологии и даёт админке возможность добавлять, редактировать и удалять записи.

Спецификация фиксирует договорённости по данным, API и поведению интерфейса. Если код меняется, этот документ нужно обновлять вместе с ним.

Пошаговая инструкция для ручного ввода данных через админку: [manual-entry/README.md](manual-entry/README.md).

## Текущая Архитектура

- `server.py` запускает локальный HTTP-сервер и API.
- `data/archive.sqlite3` хранит основную базу данных SQLite.
- `data/events.csv` используется как источник первичного импорта при создании пустой БД.
- `data/timeline_data.json` и `data/scale_data.json` автоматически пересобираются из SQLite для TimelineJS и шкалы.
- `migrations/*.sql` хранит историю изменений схемы БД.
- `index.html` показывает хронологию из JSON-файлов, собранных сервером из БД.
- `admin.html` управляет событиями, пользователями, справочниками, аудитом и структурой БД через API.
- `js/admin.js` отвечает за интерфейс админки, включая модальный предпросмотр публичной карточки события.
- `js/date-format.js` (`ArchiveDateFormat`) — общая логика форматирования дат: точность, ≈, диапазон начало—окончание; используется в админке (подсказка, предпросмотр) и на главной странице.
- `js/date-search.js` отвечает за поиск по дате и отображение выбранного факта на главной странице.
- `scripts/convert_csv_to_timelinejson.py` — устаревший автономный скрипт; **не использовать** для пересборки JSON. Актуальный путь — `server.py` → `events_to_timeline()` / `write_timeline_files()`.

## Поток Данных

Основной источник данных: `data/archive.sqlite3`.

Поток для публичной страницы:

1. Сервер запускается командой `python server.py`.
2. Сервер применяет миграции.
3. Если БД пустая, сервер импортирует начальные события из `data/events.csv`.
4. Сервер пересобирает `data/timeline_data.json` и `data/scale_data.json` из SQLite.
5. В публичные JSON-файлы попадают только события с `status = published` и `verification_status = verified` (фильтр `is_event_public_ready()` в `events_to_timeline()` и `events_to_scale()`; то же правило в админке — `isEventPublicReady` в `js/admin.js`).
6. `index.html` передаёт `data/timeline_data.json` в TimelineJS.
7. Поиск по дате использует тот же `data/timeline_data.json`.

Поток для админки:

1. Пользователь входит через `POST /api/auth/login`; сессия хранится в cookie `archive_session`.
2. Админка загружает события через `GET /api/events`, справочники через `/api/sources`, `/api/media`, `/api/tags`, `/api/groups`.
3. Сохранение события идёт через `POST /api/events`.
4. Удаление события идёт через `DELETE /api/events/:id`.
5. После сохранения или удаления сервер обновляет SQLite и пишет запись в `audit_log`.
6. После изменения событий сервер автоматически пересобирает JSON-файлы для публичной страницы.
7. Администратор может просматривать последние изменения через `GET /api/audit`.

Важно: TimelineJS стабильно работает со статическим JSON-файлом `data/timeline_data.json`. Поэтому БД не отдаётся в TimelineJS напрямую, а сначала преобразуется в совместимый JSON.

### Связи Событий

- `event_sources` связывает события с источниками.
- `event_media` связывает события с медиа.
- `event_tags` связывает события с тегами.
- `event_groups` связывает события с группами.

`GET /api/events` отдаёт для каждого события массивы `source_ids`, `media_ids`, `tag_ids`, `group_ids`, а также развёрнутые `source_items`, `media_items`, `tag_items`, `group_items`.

`POST /api/events` принимает эти массивы и сохраняет связи в `event_sources`, `event_media`, `event_tags`, `event_groups`.

`data/timeline_data.json` включает связанные записи в служебных полях `_sources`, `_media_items`, `_tag_items`, `_group_items`, а также атрибуты фазы 1 в полях `_hashtag`, `_summary`, `_event_type`, `_scale`, `_domain`, `_category`, `_subcategory`, `_country_name`, `_region`, `_city`, `_verification_status`, `_start_date_precision`, `_end_date_precision`, `_start_date_approximate`, `_end_date_approximate`, `_related_events`.

Публичная карточка события показывает связанные источники, медиа и теги под описанием события.

Поля `media_url`, `media_caption`, `media_credit` и строковое поле `tags` в `events` пока сохраняются для совместимости с TimelineJS.

## Миграции БД

Миграции лежат в папке `migrations`.

Текущие миграции:

```text
migrations/001_create_events.sql
migrations/002_add_event_status.sql
migrations/003_create_users_and_sessions.sql
migrations/004_create_reference_tables.sql
migrations/005_create_audit_log.sql
migrations/006_create_groups.sql
migrations/007_add_phase1_attributes.sql
migrations/008_rename_slug_to_hashtag.sql
migrations/009_split_date_approximate.sql
migrations/010_split_date_precision.sql
```

При запуске `server.py` сервер:

1. Создаёт служебную таблицу `schema_migrations`, если её ещё нет.
2. Смотрит, какие миграции уже применены.
3. Берёт все файлы `*.sql` из папки `migrations` по порядку имени.
4. Применяет только новые миграции.
5. Записывает применённую миграцию в `schema_migrations`.

Чтобы изменить схему БД, нужно добавлять новый файл, например:

```text
migrations/002_add_users_table.sql
```

Старые миграции после применения лучше не редактировать: они являются историей схемы.

## API

Базовый адрес при локальном запуске:

```text
http://127.0.0.1:8000
```

### `GET /api/events`

Возвращает список событий в формате, удобном для админки.

Требует входа в систему.

Ответ:

```json
{
  "events": []
}
```

### `POST /api/events`

Создаёт или обновляет событие по полю `id`.

Требует роль `admin` или `editor`.

Минимальные обязательные поля:

```json
{
  "id": "ev-0001",
  "hashtag": "ДеньПобеды",
  "start_year": "1945",
  "start_month": "5",
  "start_day": "9",
  "start_date_precision": "day",
  "end_date_precision": "",
  "headline": "День Победы",
  "summary": "Окончание Великой Отечественной войны в Европе.",
  "verification_status": "verified",
  "event_type": "праздник",
  "scale": "international",
  "domain": "политика",
  "country_name": "Россия",
  "related_events": "ev-0002",
  "source_ids": ["src-0001"],
  "media_ids": ["med-0001"],
  "tag_ids": ["tag-0001"],
  "group_ids": ["grp-0001"]
}
```

### `DELETE /api/events/:id`

Удаляет событие по его `id`.

Требует роль `admin` или `editor`.

Ответ:

```json
{
  "deleted": true
}
```

### `GET /api/timeline` и `GET /api/timeline.json`

Возвращает события в формате TimelineJS.

Это служебный API-формат. Публичная страница сейчас использует статический файл `data/timeline_data.json`, потому что локальная версия TimelineJS надёжнее работает с обычным JSON-файлом.

### `GET /api/scale` и `GET /api/scale.json`

Возвращает компактные данные для шкалы:

- `minYear`
- `maxYear`
- `events`

Публичная страница сейчас использует статический файл `data/scale_data.json`, который сервер пересобирает из SQLite.

### `GET /api/db/schema`

Возвращает структуру SQLite-базы для раздела “База данных” в админке:

- список таблиц;
- поля каждой таблицы;
- типы полей;
- признаки `NOT NULL` и `PRIMARY KEY`;
- значения по умолчанию;
- индексы;
- количество записей.

Требует роль `admin`.

### `POST /api/auth/login`

Проверяет email и пароль на сервере. При успешном входе создаёт серверную сессию и выставляет cookie `archive_session`.

### `POST /api/auth/logout`

Удаляет текущую серверную сессию и очищает cookie `archive_session`.

### `GET /api/auth/session`

Возвращает текущего пользователя по cookie-сессии или `null`, если пользователь не вошёл.

### `GET /api/users`

Возвращает список пользователей для админки.

Требует роль `admin`.

### `POST /api/users`

Создаёт или обновляет пользователя. Пароль хранится только в виде PBKDF2-хеша.

Требует роль `admin`.

### `DELETE /api/users/:id`

Удаляет пользователя.

Требует роль `admin`. Текущий пользователь не может удалить сам себя.

### `GET /api/sources`, `GET /api/media`, `GET /api/tags`, `GET /api/groups`

Возвращают справочники источников, медиа, тегов и групп.

Требуют роль `admin` или `editor`.

### `POST /api/sources`, `POST /api/media`, `POST /api/tags`, `POST /api/groups`

Создают или обновляют запись справочника по полю `id`.

Требуют роль `admin` или `editor`.

### `DELETE /api/sources/:id`, `DELETE /api/media/:id`, `DELETE /api/tags/:id`, `DELETE /api/groups/:id`

Удаляют запись справочника.

Требуют роль `admin` или `editor`.

### `POST /api/media/upload`

Загружает файл изображения в `data/uploads/` (multipart/form-data, поле `file`).

Требует роль `admin` или `editor`.

Ответ:

```json
{
  "item": {
    "filename": "photo_abc123.jpg",
    "url": "/data/uploads/photo_abc123.jpg"
  }
}
```

Загрузка используется в админке при выборе медиа из папки; запись в справочник `media` создаётся отдельно через `POST /api/media`.

### `GET /api/audit`

Возвращает последние 100 записей аудита изменений.

Требует роль `admin`.

Ответ:

```json
{
  "audit": [
    {
      "id": 1,
      "actor_user_id": "usr-001",
      "actor_email": "admin@archive.local",
      "action": "update",
      "entity_type": "event",
      "entity_id": "ev-0001",
      "summary": "День Победы",
      "created_at": "2026-06-08 12:00:00"
    }
  ]
}
```

Записи аудита создаются автоматически при `create`, `update` и `delete` для событий, пользователей, источников, медиа, тегов и групп.

## Правила Данных

- `id` должен быть стабильным: после публикации его нельзя менять без причины.
- `start_year` обязателен для каждого события.
- `headline` обязателен для каждого события.
- Если известен только год, `start_month` и `start_day` остаются пустыми.
- Для длительных событий используются поля `end_year`, `end_month`, `end_day`.
- Теги можно писать через точку с запятой: `СССР;Праздники`.
- Допустимые статусы публикации: `draft`, `review`, `published`.
- Допустимые статусы проверки: `verified`, `needs_review`, `disputed`, `unconfirmed`.
- **Связь статусов:** на главную попадают только события с `status = published` и `verification_status = verified` — при сохранении через API, в админке, предпросмотре и при экспорте JSON (`is_event_public_ready()` / `isEventPublicReady`).
- При `status = published` и пустой проверке сервер автоматически ставит `verification_status = verified`.
- При `status = published` и проверке `needs_review`, `disputed` или `unconfirmed` сохранение через API отклоняется.
- При `status = review` без проверки сервер ставит `verification_status = needs_review`.
- При `status = draft` без проверки сервер ставит `verification_status = unconfirmed`.
- `hashtag` — короткая метка без пробелов; `#` при вводе необязателен, в интерфейсе показывается с `#`.
- Допустимые значения `start_date_precision` и `end_date_precision`: `day`, `month`, `year`, `approximate`; пустое значение = **Авто**.
- **Авто (начало):** день+месяц+год → `day`; месяц+год → `month`; только год → `year`.
- **Авто (окончание):** по тем же правилам для полей окончания.
- Если задан год окончания, на карточке показывается диапазон `начало — окончание` с отдельной точностью для каждой стороны.
- **≈ Приблизительная** добавляет префикс `≈` к соответствующей дате.
- Точность `approximate` автоматически включает чекбокс ≈ на той же стороне.
- Публичная карточка и предпросмотр форматируют дату через `js/date-format.js` (`ArchiveDateFormat.formatEventDate`).
- Допустимые значения `scale`: `local`, `national`, `regional`, `international`.
- `hashtag` события должен быть уникальным, если задан.
- `related_events` хранит коды событий через `;` или `,`.
- В публичный таймлайн и шкалу попадают только события с `published` + `verified`.
- Первичный импорт из `data/events.csv` (`import_csv`) проходит `clean_event()` — те же правила дат, статусов и хештега, что и API. Невалидные строки пропускаются с сообщением в консоль.
- Заголовки `data/events.csv` совпадают с `EVENT_COLUMNS` (32 поля). Инструкция: [manual-entry/README.md](manual-entry/README.md).
- Допустимые роли пользователей: `admin`, `editor`, `viewer`.
- Пароли пользователей не хранятся в открытом виде.
- Изменять события через API могут только `admin` и `editor`.
- Управлять пользователями может только `admin`.
- Создание, изменение и удаление событий, пользователей, источников, медиа и тегов фиксируется в `audit_log`.
- CSV больше не считается основной БД, он нужен только для импорта или резервной копии.
- `data/timeline_data.json` и `data/scale_data.json` не редактируются вручную: они генерируются из SQLite.

## Админка

Разделы админки:

- **Наполнение** — события.
- **Пользователи** — только для `admin`.
- **Источники**, **Медиа**, **Теги**, **Группы** — справочники для `admin` и `editor`.
- **Аудит** — последние изменения, только для `admin`.
- **База данных** — просмотр схемы SQLite, только для `admin`.

Админка должна:

- Загружать события из `/api/events`.
- Сохранять событие через `POST /api/events`.
- Удалять событие через `DELETE /api/events/:id`.
- Позволять выбрать статус публикации: черновик, на проверке или опубликовано.
- Позволять задавать атрибуты фазы 1: `hashtag`, `summary`, классификацию, географию, точность даты, статус проверки и связанные события.
- Позволять выбрать связанные источники, медиа, теги и группы из справочников в форме события.
- Позволять указывать `evidence_quote` в форме источника.
- Давать поиск, счётчик выбранного и быстрый переход к созданию записи при выборе справочников в форме события.
- Показывать модальный предпросмотр публичной карточки события поверх админки.
- Давать возможность скачать CSV-копию текущих данных.
- Показывать структуру БД через раздел “База данных”.
- Показывать последние записи аудита в разделе “Аудит”.
- После сохранения или удаления события публичный таймлайн должен получать обновлённые JSON-файлы.
- Выполнять вход через серверный endpoint, а не через `localStorage`.
- Управлять пользователями через SQLite и серверный API.
- Управлять источниками, медиа, тегами и группами через отдельные разделы админки.

### Форма события (вёрстка)

Компактная верхняя строка `.admin-event-meta-bar` (на всю ширину):

- **Код** (`event-id`, только чтение, автогенерация);
- **Публикация** (`event-status`: черновик / на проверке / опубликовано);
- **Проверка** (`event-verification-status`);
- **Важн.** (`event-importance`, 1–10);
- **Хештег** (`event-hashtag`).

Подсказка `#event-status-hint` объясняет, попадёт ли событие на главную.

Блок дат `.admin-date-range` — две колонки **ДАТА НАЧАЛА** и **ДАТА ОКОНЧАНИЯ**:

| Строка | Содержимое | Размеры полей |
| --- | --- | --- |
| 1-я (`.admin-date-values`) | День, Месяц, Год | 55px, 75px, 100px |
| 2-я (`.admin-date-options`) | ≈ Приблизительная, селект точности | чекбокс + селект 92px |

Подсказка `#event-date-hint` показывает итоговую дату по правилам `ArchiveDateFormat`.

Связи со справочниками выбираются через модальные меню **+ Источник / + Медиа / + Тег / + Группа** (Новый / База / Папка для медиа). Строковое поле `tags` в форме не редактируется вручную — собирается из выбранных тегов. Поля `media_url`, `media_caption`, `media_credit` скрыты в форме; основное медиа задаётся через справочник или загрузку файла.

Стили формы: `css/admin.css` (классы `admin-date-*`, `admin-event-meta-bar`).

### Предпросмотр события

Кнопка **«Предпросмотр карточки»** в разделе “Наполнение” открывает модальное окно поверх админки.

Предпросмотр повторяет публичный блок факта с главной страницы:

- заголовок и хештег;
- дату (с точностью и ≈), место, группу и краткие метаданные (`scale`, `event_type`, `domain`);
- краткое (`summary`) и полное описание;
- основное медиа события (`media_url` → блок `media` в JSON);
- связанные источники (с `evidence_quote`), медиа и теги из выбранных справочников.

Поведение:

- пока окно открыто, содержимое обновляется при изменении полей формы;
- предупреждение, если событие **не готово** к главной: `draft`, `review`, или `published` без `verified`;
- закрытие: кнопка **«Закрыть»**, клик по затемнённому фону или клавиша `Esc`.

### Публичная карточка события

На главной странице блок `.fact-panel` (`js/date-search.js`) показывает выбранное событие:

| Показывается | Поля JSON / БД |
| --- | --- |
| Заголовок | `text.headline` |
| Хештег | `_hashtag` |
| Дата | `start_date`, `end_date`, `_start_date_precision`, `_end_date_precision`, `_start_date_approximate`, `_end_date_approximate` через `ArchiveDateFormat` |
| Место | `_city`, `_region`, `_country_name` |
| Метаданные | `group`, `_scale`, `_event_type`, `_domain` |
| Краткое описание | `_summary` — отдельный блок, всегда целиком |
| Полное описание | `text.text` — свёрнуто до 3 строк, кнопки «Читать полностью» / «Свернуть» |
| Основное медиа | `media` — внизу блока полного описания, видно только в развёрнутом состоянии |
| Источники | `_sources` (+ `evidence_quote`) |
| Связанные медиа | `_media_items` |
| Теги | `_tag_items` |

**Есть в JSON, но не выводится на карточке:** `_category`, `_subcategory`, `_verification_status`, `_related_events`, `_group_items` (используется только строка `group`).

### 6. Убрать дублирование legacy-полей в `events`

**Суть:** в таблице `events` одновременно хранятся строковые `tags`, `media_url`, `media_caption`, `media_credit` и нормализованные связи через `event_tags`, `event_media`.
