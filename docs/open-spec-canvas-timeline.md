# Open Spec: Canvas + API — новый таймлайн

## Связь с основной спецификацией

Этот документ описывает **замену TimelineJS** на собственный рендер (HTML5 Canvas) и **viewport-ориентированный API**.
Общая архитектура проекта, схема БД, админка и правила данных — в [open-spec.md](open-spec.md).

Детальные решения по режиму **сравнения стран** (полосы-страны, флажки-даты, лестница зума, интервальные флаги, поведение поиска по дате) — в [open-spec-country-lanes.md](open-spec-country-lanes.md); порядок работ там привязан к этапам этого документа.

Если реализация меняет контракт API или поведение главной страницы, обновлять **оба** документа.

## Цель

1. Убрать зависимость от TimelineJS и хрупкие патчи в `js/timeline-layout.js`.
2. Обеспечить **стабильную** шкалу времени при загрузке, скролле страницы и изменении размера окна — без гонок инициализации.
3. Подготовить архитектуру к **масштабу до 10 000 000** опубликованных событий, **настраиваемым полосам флагов**, **наложению по приоритету** и **фильтрам** по атрибутам.
4. Держать бюджет производительности явным с первого дня: 60 fps при pan/zoom, отклик API в холодном и тёплом кэше, отсутствие лишней перерисовки.
5. Сохранить текущий UX главной страницы: menubar слева, поиск по дате (включая поиск ближайшего события), блок факта (`.fact-panel`), форматирование дат через `ArchiveDateFormat`, доступность с клавиатуры.

## Проблема текущего решения

| Симптом | Причина |
| --- | --- |
| Шкала годов пропадает после скролла страницы | TimelineJS пересчитывает layout DOM; glue-код (`updateDisplay`, `ResizeObserver`) гоняется с внутренней логикой библиотеки |
| Шкала/флажки не появляются до resize окна | `updateDisplay()` вызывается до того, как контейнер получил ненулевые размеры |
| Превью медиа на маркерах | TimelineJS рисует `.tl-timemarker-media-container`; CSS/JS-патчи нестабильны |
| Невозможность масштаба 10M | Весь `timeline_data.json` загружается в браузер; каждое событие — DOM-узел |

**Вывод:** TimelineJS подходит для небольших статических историй, но не для целевых требований архива. Дальнейшее наращивание `timeline-layout.js` не рекомендуется.

### Уроки для нового рендера (чтобы не повторить те же классы багов)

Собственный рендер устраняет зависимость от чужого layout-движка, но **гонка инициализации** (симптом №2) — это баг класса «отрисовка раньше, чем контейнер получил размеры», а не баг конкретно TimelineJS. Его легко воспроизвести и в новом коде, если не заложить защиту явно:

- `ArchiveCanvasTimeline.init()` **не должен** вызывать первый `draw()` до тех пор, пока `ResizeObserver` не сообщит о ненулевых `width`/`height` контейнера. Первый кадр рисуется из колбэка обсервера, а не из `DOMContentLoaded`/`load`.
- Если размеры уже ненулевые к моменту `init()` (частый случай), рисуем немедленно и всё равно подписываемся на `ResizeObserver` для последующих изменений — на скролл/resize реагирует один и тот же путь кода, отдельного «догоняющего» таймера/ретраев (как `MAX_LAYOUT_RETRIES` в старом `timeline-layout.js`) быть не должно.
- Тест на регресс этого класса багов входит в критерии приёмки MVP (см. ниже) и должен проверяться вручную и/или скриншот-тестом при каждом релизе рендера.

## Целевая архитектура

```text
┌─────────────────────────────────────────────────────────────┐
│  index.html                                                  │
│  ├─ js/canvas-timeline.js    — рендер, pan/zoom, hit-test   │
│  ├─ js/date-search.js        — поиск, панель факта (адапт.) │
│  └─ js/date-format.js        — без изменений                │
└──────────────────────────┬──────────────────────────────────┘
                           │ fetch (viewport + filters)
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  server.py                                                   │
│  ├─ GET /api/timeline/markers   — маркеры в окне времени    │
│  ├─ GET /api/timeline/range     — min/max год для шкалы     │
│  ├─ GET /api/timeline/nearest   — ближайшее событие к дате   │
│  └─ GET /api/timeline/event/:id — полная карточка (опц.)    │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
                    archive.sqlite3
                    (далее PostgreSQL при 10M+)
```

### Принципы

- **Сервер отдаёт только видимое:** окно `[from, to]`, уровни детализации, фильтры, лимит записей.
- **Клиент рисует на Canvas:** ось, подписи годов, флажки в одной или нескольких **полосах (lanes)**; без тысяч DOM-узлов.
- **Рендер идёт по dirty-флагу, не по таймеру.** `draw()` вызывается только когда изменилось состояние (viewport, данные, hover, resize) — постоянный цикл `requestAnimationFrame`, который рисует кадр даже без изменений, не допускается: это лишняя нагрузка на CPU/батарею без пользы.
- **Данные маркеров в клиентском кэше хранятся в typed arrays** (structure-of-arrays: отдельные `Float64Array`/`Int32Array` для даты, `importance`, индекса в таблице строк, а не массив объектов) — меньше давления на GC и лучше локальность кэша при большом числе маркеров в памяти одновременно.
- **Полная карточка события** может приходить в ответе маркера (MVP) или отдельным запросом (этап 3).
- **Статические JSON** (`timeline_data.json`, `scale_data.json`) остаются на переходный период; после cutover — deprecated или только для экспорта.

## Этапы внедрения

### Этап 1 — MVP (замена TimelineJS)

**Срок (оценка):** ~1–1.5 недели совместной работы в Cursor (расширен на 2 пункта, которые раньше планировались на этап 2 — см. ниже).

| Задача | Результат |
| --- | --- |
| API viewport | `GET /api/timeline/markers` с параметрами `from`, `to`, `limit` |
| API ближайшей даты | `GET /api/timeline/nearest?date=` — **включено в MVP**, не в этап 2 (см. «Почему нельзя откладывать») |
| Canvas-рендер | Ось времени, pan (горизонтальный скролл), базовые маркеры, dirty-flag рендер, devicePixelRatio-корректный canvas |
| Инициализация без гонки | Первый `draw()` только после подтверждения ненулевых размеров контейнера (см. «Уроки для нового рендера») |
| Один lane | Все события в одной полосе; `importance` влияет на z-order |
| Разводка перекрывающихся маркеров | Простая раскладка (жадная упаковка по рядам/микро-сдвиг) для маркеров, попадающих в одно и то же пиксельное окно — **включено в MVP**, чтобы не потерять то, что бесплатно давал TimelineJS (см. ниже) |
| Интеграция | Клик/hover → выбор события → `.fact-panel` через `date-search.js` |
| Поиск по дате | `goToDate()` прокручивает viewport к дате без `TL.Timeline`; при отсутствии точного совпадения использует `/api/timeline/nearest` |
| Доступность | Параллельное скрытое (`sr-only`) accessible-дерево маркеров текущего viewport для клавиатуры и скринридеров (см. «Доступность») |
| Тач | `touch-action` разделяет вертикальный скролл страницы и горизонтальный pan канваса (см. «Тач и мобильные жесты») |
| Вёрстка | Menubar слева, фиксированная высота блока таймлайна — как сейчас |
| Удаление legacy | См. «Стратегия миграции» — не раньше, чем все критерии приёмки MVP зелёные в проде |

**Почему поиск ближайшей даты и разводка маркеров нельзя откладывать на этап 2:**

- `js/date-search.js` уже показывает пользователю ближайшее событие с сообщением «Точного совпадения нет. Показано ближайшее (±N дн.)», если точной даты нет. Это рабочая функция сегодня — перенос на Canvas без неё был бы тихим регрессом уже в MVP, а не осознанным компромиссом.
- TimelineJS автоматически разносит маркеры, которые накладываются друг на друга по дате — без этого MVP на «десятках событий» будет выглядеть хуже прежнего сразу же, а не только при росте архива. Полноценные полосы (lanes) и бейджи «+N» — это этап 2, но **минимальная** раскладка (не дать двум флажкам залезть друг на друга) нужна с первого дня.

**Критерии приёмки MVP:**

- [ ] При первой загрузке шкала годов видна без resize окна.
- [ ] После скролла страницы вниз и обратно шкала остаётся корректной.
- [ ] Клик по маркеру обновляет `.fact-panel` с теми же полями, что сейчас.
- [ ] Поиск по дате находит событие и центрирует viewport; при отсутствии точной даты показывает ближайшее событие с тем же сообщением, что сейчас.
- [ ] Маркеры на близких датах не перекрываются полностью — видно и кликабельно каждое.
- [ ] На маркерах **нет** превью изображений — только текст/иконка флажка.
- [ ] Таймлайн проходим с клавиатуры (Tab/Enter между маркерами), скринридер озвучивает заголовок и дату выбранного события.
- [ ] На мобильном вертикальный скролл страницы работает поверх канваса; горизонтальный pan не блокирует его.
- [ ] Работает на текущем объёме данных (~десятки событий) через API, не через полный JSON.
- [ ] Рендер не перерисовывает кадр, если состояние (viewport/данные/hover) не изменилось — проверяется профайлером (нет непрерывного `requestAnimationFrame`-цикла в состоянии покоя).

**Срок (оценка):** +3–6 недель (с миграцией инфраструктуры).

- [ ] Запрос viewport при 10M строк в БД укладывается в целевой бюджет (&lt; 300 ms на сервере, тёплый кэш &lt; 50 ms).
- [ ] Браузер не загружает более N записей за один запрос (N задаётся в spec API).

## API

Базовый адрес: `http://127.0.0.1:8000` (как в [open-spec.md](open-spec.md)).

### `GET /api/timeline/markers`

Возвращает маркеры для текущего окна времени. **Основной endpoint** для Canvas-рендера.

#### Query-параметры

| Параметр | Тип | Обязательный | Описание |
| --- | --- | --- | --- |
| `from` | ISO-дата или `YYYY-MM-DD` | да* | Начало окна (включительно). *На MVP допустим только год: `from=1940` |
| `to` | ISO-дата или `YYYY-MM-DD` | да* | Конец окна (включительно) |
| `limit` | int | нет | Макс. число маркеров (default `500`, max `2000`) |
| `lanes` | JSON (URL-encoded) | нет | Конфиг активных полос (см. раздел «Полосы»); без параметра — только полоса по умолчанию |
| `zoom` | float | нет | Относительный масштаб viewport (px/год или enum `far`/`mid`/`near`) — для LOD |
| `min_importance` | int | нет | Нижняя граница важности для LOD (этап 2+) |
| `domain` | string | нет | Фильтр по `events.domain` |
| `scale` | string | нет | Фильтр по `events.scale` |
| `country_name` | string | нет | Фильтр по стране |
| `event_type` | string | нет | Фильтр по типу |
| `category` | string | нет | Фильтр по категории |
| `tag` | string | нет | Фильтр по slug/name тега |
| `cluster` | `0` \| `1` | нет | `1` — вернуть агрегаты вместо отдельных маркеров (этап 3) |

Все параметры-фильтры передаются в SQL только через параметризованные запросы (как остальной `server.py`) — конкатенация строк для `domain`/`country_name`/`tag` и т.п. запрещена (SQL injection).

#### Правила выборки

- В выборку попадают только события с `status = published` и `verification_status = verified` (как `is_event_public_ready()`).
- Сортировка для отрисовки: `lane_id` asc, затем `importance` desc, затем дата начала.
- Если событие попадает в несколько полос (несколько тегов/групп), API может вернуть **несколько маркеров** с одним `id` и разными `lane_id` (см. «Полосы»). Это уменьшает эффективный `limit` на уникальные события — см. согласование в разделе «Полосы».
- События с `end_year` занимают интервал `[start, end]` на оси и рисуются **интервальным флагом с двумя ногами** — решение принято, см. [open-spec-country-lanes.md](open-spec-country-lanes.md) п. 9 (закрывает открытый вопрос № 3). Агрегаты/кластеры считают только точечные события; интервалы возвращаются отдельно на каждом уровне зума, где их диапазон пересекает viewport.

### `GET /api/timeline/nearest`

**Часть MVP, не этапа 2.** Возвращает ближайшее к заданной дате опубликованное событие — нужна для сохранения текущей функции поиска по дате (`date-search.js`, см. «Уроки для нового рендера» и критерии приёмки MVP).

#### Query-параметры

| Параметр | Тип | Обязательный | Описание |
| --- | --- | --- | --- |
| `date` | `YYYY-MM-DD` | да | Целевая дата поиска |
| `lanes` | JSON | нет (этап 2, 2026-07-21) | Тот же формат, что и у `/api/timeline/markers` — конфиг активных полос-участников. Если задан, ответ дополнительно содержит `by_lane` с тем же результатом поиска, но посчитанным только на подмножестве событий, где страна полосы — участник (`event_countries.role='participant'`) |

#### Ответ

```json
{
  "date": "1940-01-15",
  "exact": false,
  "found": true,
  "diff_days": 0,
  "marker": {
    "id": "ev-0016",
    "headline": "Советско-финляндская война (Зимняя война)",
    "date": { "year": 1939, "month": 11, "day": 30 },
    "end_date": { "year": 1940, "month": 3, "day": 13 },
    "precision": "day",
    "approximate": false
  },
  "matches": [],
  "covering": [],
  "ongoing": [ "...тот же payload, что marker..." ],
  "nearest_prev": null,
  "nearest_next": null,
  "by_lane": {
    "lane-ussr": { "title": "СССР", "found": true, "marker": { "...": "..." }, "nearest_prev": null, "nearest_next": null }
  }
}
```

Реализация (`_nearest_search()` в `server.py`, переиспользуется для общего ответа и для каждой полосы в `by_lane`): полный проход по `events` (публичным, `attached_to` пусто) с приоритетом `exact → covering → ongoing → ближайший по модулю diff_days`; `found` — сводный признак «на дату что-то есть» (exact/covering/ongoing), отдельно от `exact` (точное совпадение день-в-день). На масштабе потребует индексации (предвычисленный столбец «день от эпохи»), сейчас — линейный проход по всем событиям, приемлемо для текущего объёма.

Требования из [open-spec-country-lanes.md](open-spec-country-lanes.md) (пп. 2–3, 7, 9) — **реализовано, 2026-07-21**:

- события с точностью «месяц»/«год»/`approximate` **покрывают** запрошенную дату (`covering`) и не проваливаются на «ближайший» фолбэк — раньше был баг: `covering`/`ongoing` игнорировались, теперь исправлено и покрыто тестом;
- интервальные события (`start ≤ date ≤ end`) возвращаются в `ongoing`, а не «пусто»;
- при активных полосах (`lanes=`) ближайшие события возвращаются **по каждой полосе отдельно** через `by_lane` с расстоянием в днях (`diff_days`), не только по архиву целиком.

Клиент: `js/canvas-timeline.js` (`goToDate()`) и `js/date-search.js` (canvas-режим, определяется по наличию `setLanes`/`goToDate` у объекта таймлайна) — при находке центрирует viewport и выделяет реальный маркер; при отсутствии — пунктирный якорь на запрошенной дате + панель «ближайших» по каждой активной полосе. Автозум именно до уровня «месяц» (лестница зума, п. 8) — не сделан, центрирование пока не привязано к дискретным уровням.

## Производительность и рендер (`js/canvas-timeline.js`)

Раздел фиксирует конкретные технические решения, а не только намерение «работать быстро».

### Клиент

| Приём | Зачем |
| --- | --- |
| Рендер по dirty-флагу | `draw()` вызывается только при изменении viewport/данных/hover/resize; в состоянии покоя нет активного `requestAnimationFrame`-цикла — не тратим CPU/батарею впустую |
| `devicePixelRatio`-корректный backing store | Canvas-буфер выставляется в `CSS-размер × devicePixelRatio`, контекст масштabируется через `ctx.scale(dpr, dpr)` — чёткий текст без избыточного оверрендера на HiDPI |
| Structure-of-arrays для кэша маркеров | Даты, `importance`, координаты — в `Float64Array`/`Int32Array`, не в массиве объектов; меньше давления на GC, лучше локальность при бинарном поиске по времени |
| Бинарный поиск границ viewport | Маркеры в клиентском кэше отсортированы по дате; видимый диапазон вычисляется `O(log n)` по границам, а не полным проходом массива на каждый кадр |
| Отмена устаревших запросов | Каждый `fetch` к `/api/timeline/markers` — через `AbortController`; при быстром pan/zoom предыдущий незавершённый запрос отменяется, применяется только последний |
| Дебаунс запросов при непрерывном pan/zoom | Fetch к серверу шлётся не на каждое событие wheel/drag, а с задержкой ~50–100 мс после того, как жест «устоялся»; сам рендер (перерисовка по локальному кэшу) остаётся мгновенным |
| Web Worker для hit-test/кластеризации | При высокой плотности маркеров (этап 3) вычисления, не связанные с рисованием, уносятся в worker; данные передаются как transferable typed arrays, без копирования |

### Сервер

| Приём | Зачем |
| --- | --- |
| Параметризованные SQL-запросы везде | Защита от SQL injection при фильтрах (`domain`, `tag`, `country_name` и т.д.) |
| Составные индексы под фильтры viewport | `(start_year, start_month, start_day)` + индексы на поля фильтров и `importance` — без них `/markers` деградирует линейно с ростом БД |
| `Cache-Control`/`ETag` на редко меняющихся ответах | `/range` и повторяющиеся `/markers`-запросы с одинаковой сигнатурой не пересчитываются заново |
| Жёсткий `limit` с ясным откликом `truncated` | Клиент всегда знает, что показана не вся выборка, и может сузить окно/фильтр |
| Пул потоков/asyncio вместо однопоточной обработки | Нужно уже на этапе 3 при конкурентных запросах — миграция на PostgreSQL сама по себе не даёт пропускной способности без этого |

### Бюджет производительности (ориентиры, уточняются после первых замеров)

### Слои отрисовки (снизу вверх)

1. **Фон** — цвет `--tl-background` / существующие CSS-переменные.

### Тач и мобильные жесты

- Канвасу задаётся `touch-action: pan-y` по умолчанию: одно вертикальное касание/свайп проходит как обычный скролл страницы, не перехватывается таймлайном.

## Доступность (a11y)

TimelineJS даёт keyboard-навигацию и ARIA-разметку по маркерам «из коробки»; Canvas рисует пиксели и по умолчанию ничего из этого не предоставляет. Без отдельного плана переход на Canvas — это регресс доступности, а не нейтральное изменение.

**Решение:** параллельное скрытое (`sr-only`, визуально не показывается, но доступно скринридеру) accessible-дерево, синхронизированное с текущим viewport:

- На каждый маркер текущего viewport — элемент списка с `role="button"`, `tabindex`, доступным именем `"{headline}, {дата через ArchiveDateFormat}"`.
- Roving `tabindex`: Tab заходит в таймлайн один раз, дальше стрелки/Tab двигают фокус между маркерами.
- Активация (Enter/Space) вызывает тот же `select`, что и click по канвасу — выбор события и обновление `.fact-panel`.
- Список пересобирается при каждом обновлении маркеров (pan/zoom/фильтр), но не чаще dirty-рендера канваса.
- Сам `<canvas>` получает `aria-hidden="true"` или `role="img"` с кратким описанием — управление доступностью идёт через параллельный список, а не через попытки озвучить пиксели.

Критерий приёмки MVP: таймлайн проходим с клавиатуры, скринридер озвучивает заголовок и дату выбранного события (см. «Этап 1»).

## Полосы (lanes)

### Как понимается задача (зафиксировано по уточнению)

1. **По умолчанию — одна полоса.** На ней показываются все опубликованные события текущего viewport (без разбиения по атрибутам).
2. **Пользователь может добавить полосы** через UI на главной странице (не через админку).
3. **При добавлении полосы** пользователь выбирает **измерение группировки** — одно из четырёх семейств атрибутов события. Маркеры на новых полосах **назначаются автоматически** по уже заполненным полям события в БД; вручную привязывать событие к полосе не нужно.
4. **Внутри одной полосы** на одной дате несколько событий накладываются с учётом **`importance`** (приоритет).

Отдельного поля `timeline_level` в БД **не требуется** — полосы определяются конфигом UI и метаданными события.

### Четыре семейства атрибутов

| Семейство | Поля / связи в БД | Пример значения полосы |
| --- | --- | --- |
| **Группы** | `events.group`, `group_items` / `event_groups`; иерархия `groups.parent_id` (миграция 013) | «Политика», `grp-0003`; полоса по основной группе агрегирует её подгруппы ([open-spec-country-lanes.md](open-spec-country-lanes.md) п. 6) |
| **Теги** | `events.tags`, `tag_items` / `event_tags` | «СССР», `tag-0003` |
| **География** | `country_name`, `region`, `city` | «Россия», «Москва» |
| **Классификация** | `domain`, `category`, `subcategory`, `event_type`, `scale` | «политика», `international` |

При добавлении полосы пользователь выбирает **семейство** и при необходимости **конкретное поле** (например, география → только `country_name`) и **значение** (или «все значения» — тогда создаётся подполоса на каждое distinct-значение в viewport).

**География и сравнение стран — блокирующая зависимость:** `country_name` хранит *место* события, а полосы сравнения стран должны строиться по *участникам* (нападение Германии на СССР должно попадать в обе полосы). Принято решение ввести M:N `event_countries` с ролью «место/участник» и нормализовать составные значения («Литва, Латвия, Эстония») — см. [open-spec-country-lanes.md](open-spec-country-lanes.md) п. 5. Миграцию выполнить **до этапа 2**.

### Согласование дублирования маркеров с лимитом (важно для масштаба)

Событие с несколькими тегами/группами может дать несколько маркеров (один `id`, разные `lane_id`) — см. «Правила выборки» в API. Это означает, что **включение нескольких полос напрямую снижает** эффективное число уникальных событий, помещающихся в `limit` одного запроса.

Правила, чтобы это не стало сюрпризом на масштабе:

- Полоса **«Все события»** (`id: default`) никогда не дублирует маркеры — она остаётся «дешёвым» базовым режимом просмотра.
- Число одновременно видимых пользовательских полос ограничено константой (ориентир: 12) — верхняя граница множителя дублирования известна заранее и учитывается при расчёте `limit`/производительности этапа 3.
- Ответ `/markers` при `truncated: true` даёт клиенту понять, что часть маркеров не показана — UI должен предлагать сузить фильтр или число полос, а не молча резать данные.

### Поведение по умолчанию при добавлении полосы

```text
Состояние A (начальное)          Состояние B (пользователь добавил «Теги»)
┌─────────────────────────┐      ┌─ Все события ─────────────────────┐
│  Все события            │  →   ├─ #СССР ──────────────────────────┤
│  ● ●  ●    ●            │      ├─ #Праздники ─────────────────────┤
└─────────────────────────┘      └─ #Космос ────────────────────────┘
```

- Полоса **«Все события»** (`id: default`) остаётся, если пользователь её не скрыл.
- Добавленная группировка **по тегам** порождает **отдельную горизонтальную полосу на каждый тег**, который встречается у событий в текущем наборе (или у всех событий архива — см. открытый вопрос № 1).
- Событие с **несколькими тегами** может дать **несколько маркеров** (один `id`, разные `lane_id`) — по одному на каждую релевантную полосу.

### Конфиг полосы (JSON)

```json
{
  "id": "lane-tag-sssr",
  "title": "СССР",
  "kind": "tag",
  "field": "tag",
  "value": "tag-0003",
  "visible": true
}
```

Типы `kind`: `default` | `group` | `tag` | `geography` | `classification`.

Для `geography` и `classification` в `field` указывается имя поля (`country_name`, `domain`, …).

### UI (этап 2)

- Кнопка **«+ Полоса»** в menubar таймлайна.
- Мастер: семейство → поле (если нужно) → режим:
  - **«Одна полоса — одно значение»** (пользователь выбирает значение из справочника / autocomplete);
  - **«Развернуть все значения»** (автоматически создать полосу на каждое distinct-значение).
- Список активных полос: показать / скрыть / удалить / изменить порядок (drag).
- Конфиг сохраняется в `localStorage` (`archive_timeline_lanes`).

### Наложение внутри одной полосы

1. Маркеры с одной календарной датой группируются.
2. Сортировка внутри группы: `importance` desc.
3. Если ширина группы &gt; порога (px):
   - **режим A:** горизонтальный сдвиг маркеров (уже в MVP как минимальная раскладка);
   - **режим B:** один маркер + badge «+N» (клик открывает список) — этап 2.

## LOD (Level of Detail)

**Принята дискретная «лестница уровней»** (век → половина века → год → месяц) с агрегатными флажками по единицам оси и спуском по клику — см. [open-spec-country-lanes.md](open-spec-country-lanes.md) п. 8. Выше уровня «месяц» флажки — всегда агрегаты «×N»; коллизии соседних дат исчезают по построению. Серверная агрегация этапа 3 (`cluster=1`) отдаёт именно эти единицы (месяц/год/полвека × полоса × count).

Пороги `importance` ниже действуют **внутри уровня «месяц»** (что показывать первым при плотных днях) и как параметр `min_importance` API; переключение самих уровней — по порогу «ширина единицы оси < ширины флажка».

| Zoom (px/год) | Поведение |
| --- | --- |
| &lt; 2 | Только `importance ≥ 8`; остальное — кластеры по годам |
| 2 – 20 | `importance ≥ 4` |
| &gt; 20 | Все события в viewport; без кластеров |

Пороги — настраиваемые константы в `canvas-timeline.js` и параметр `min_importance` в API.

## Интеграция с `date-search.js`

### Текущая зависимость

`date-search.js` использует `window.__timeline` (TimelineJS): `goToId()`, загрузка полного списка из JSON, собственный клиентский перебор для поиска ближайшей даты.

### Целевая зависимость

| Было | Станет |
| --- | --- |
| `window.__timeline` | `window.__canvasTimeline` (`ArchiveCanvasTimeline`) |
| Полный массив `events` из JSON | Кэш последнего viewport (typed arrays) + `/api/timeline/range` для границ шкалы |
| `goToId(id)` | `ArchiveCanvasTimeline.setViewport(...)` + `select(id)` |
| Клиентский перебор для поиска ближайшей даты | `GET /api/timeline/nearest?date=` — один индексированный запрос вместо перебора всего массива в браузере (быстрее уже на текущем объёме и единственный вариант, который не деградирует на масштабе) |

Функции `formatDateRu`, рендер `.fact-panel` и `ArchiveDateFormat` **не меняют контракт** — меняется только источник объекта события.

## Стратегия миграции

Легаси и новый рендер **не живут в проде параллельно дольше, чем нужно для проверки MVP**. Явная последовательность, без дублирующих путей:

1. **Разработка за query-флагом.** На время работы над MVP главная страница доступна и как `index.html` (TimelineJS, как сейчас), и как `?timeline=canvas` (или отдельная `index-canvas.html`) — только для внутреннего тестирования, не для конечных пользователей.
2. **MVP на API.** Canvas-версия сразу работает через `/api/timeline/markers` и `/nearest`; статические JSON-файлы ещё генерируются для отладки/сравнения, но canvas-путь их не читает.
3. **Регрессия UX по чеклисту** (критерии приёмки MVP выше): загрузка, скролл страницы, resize, поиск по дате (точный + ближайший), клик по маркеру, клавиатура/скринридер, тач на мобильном, хештег в карточке.
4. **Cutover.** Когда все критерии приёмки MVP зелёные — `?timeline=canvas` становится единственным поведением `index.html`, query-флаг и параллельный путь удаляются в том же коммите. TimelineJS, `timeline-layout.js` и vendor-файлы удаляются сразу после, не «когда-нибудь потом».
5. **Этап 2–3** — без отката на TimelineJS.

## Тестирование

Раньше не было отдельной стратегии — фиксируем минимальный набор, соразмерный риску (математика pan/zoom/hit-test — источник тонких багов, которые сложно поймать вручную):

| Уровень | Что покрывает |
| --- | --- |
| Юнит-тесты (чистые функции) | Бинарный поиск границ viewport, вычисление hit-test координата → маркер, вычисление раскладки при наложении, LOD-пороги — всё это чистые функции, тестируемые без DOM/canvas |
| Интеграционные тесты API | `/api/timeline/markers`, `/range`, `/nearest` на фикстурах БД: проверка `is_event_public_ready()`-фильтра, лимитов, дублирования маркеров при lanes, параметризации фильтров |
| Ручной регрессионный чеклист | Критерии приёмки MVP (см. «Этап 1») перед каждым cutover-релизом |
| Скриншот-регрессия (опционально) | Проект уже использует Playwright (`scripts/capture_manual_entry_screenshots.py`) — можно переиспользовать для сравнения скриншотов таймлайна после/до изменений рендера |
